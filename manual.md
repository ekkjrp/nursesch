## 1. 시작하기

### 1.1 설치 및 실행

```bash
# 1. 의존성 설치
npm install

# 2. 데이터베이스 초기화
cd backend
npx prisma generate
npx prisma db push
npx tsx src/seed.ts
cd ..

# 3. 개발 서버 실행 (백엔드 + 프론트엔드 동시 실행)
npm run dev
```

실행 후 브라우저에서 **http://localhost:5173** 으로 접속합니다.

> **주의:** `npm run dev` 명령은 반드시 **프로젝트 루트 폴더** (`nursesch/`)에서 실행해야 합니다.
> 이 명령은 백엔드 서버(port 3000)와 프론트엔드 서버(port 5173)를 **동시에** 실행합니다.
> 하나만 실행하면 서비스가 정상 동작하지 않습니다.

### 1.2 서버 구성

NurseSch는 **2개의 서버**로 구성됩니다. 두 서버가 모두 실행 중이어야 정상 동작합니다.

| 서버 | 포트 | 역할 | 접속 URL |
|------|------|------|---------|
| **프론트엔드** (Vite) | 5173 | 사용자 화면 (React) | http://localhost:5173 (브라우저 접속) |
| **백엔드** (Express) | 3000 | REST API 서버 | http://localhost:3000/api/* (자동 연결) |

> 사용자는 항상 **http://localhost:5173** 으로 접속합니다.
> 프론트엔드가 백엔드 API를 자동으로 프록시(중계)하므로 백엔드 URL에 직접 접속할 필요가 없습니다.

#### 개별 실행 (선택)

터미널 2개를 열어 각각 실행하는 것도 가능합니다:

```bash
# 터미널 1 — 백엔드
npm run dev:backend

# 터미널 2 — 프론트엔드
npm run dev:frontend
```

#### 정상 실행 확인

서버가 올바르게 실행되면 콘솔에 다음 메시지가 표시됩니다:

```
VITE v6.x.x  ready in xxx ms
  ➜  Local:   http://localhost:5173/
NurseSch backend running on port 3000
```

두 메시지가 모두 표시되어야 합니다.

### 1.3 초기 계정 정보

시드 데이터로 생성되는 기본 계정입니다.

| 구분 | 이메일 | 비밀번호 | 역할 |
|------|--------|---------|------|
| 관리자 (수간호사) | `admin@hospital.com` | `password123` | ADMIN |
| 간호사 1 | `nurse1@hospital.com` | `password123` | NURSE |
| 간호사 2 | `nurse2@hospital.com` | `password123` | NURSE |
| ... | `nurse3~9@hospital.com` | `password123` | NURSE |

- 관리자 계정은 **김수간** (내과 5병동 수간호사)으로 등록되어 있습니다.
- 간호사는 총 **9명**이 사전 등록되어 있습니다.

## 2. 로그인

1. 브라우저에서 **http://localhost:5173** 에 접속하면 **로그인 화면**이 표시됩니다.
   - 보라색 그라데이션 배경에 흰색 로그인 카드가 표시됩니다.
   - "NurseSch" 로고와 "간호사 근무표 자동 생성 서비스" 부제목이 보입니다.
2. 이메일과 비밀번호를 입력한 뒤 **로그인** 버튼을 클릭합니다.
3. 로그인에 성공하면 **대시보드**로 이동하며, 역할에 따라 다른 메뉴가 표시됩니다.
   - **관리자**: 대시보드, 간호사 관리, 근무 규칙, 근무표, 근무 요청
   - **간호사**: 대시보드, 근무표, 근무 요청

> 로그아웃하려면 좌측 사이드바 하단의 사용자 이름 아래 **로그아웃** 버튼을 클릭합니다.

로그인 후 화면은 크게 두 영역으로 나뉩니다.

```

│          │                                │
│ 사이드바  │         메인 콘텐츠 영역         │
│          │                                │
│ - 로고    │   (선택한 메뉴의 내용 표시)      │
```

**사이드바 메뉴:**

| 메뉴 | 관리자 | 간호사 | 설명 |

## 4. 관리자 기능

### 4.1 대시보드

로그인 직후 표시되는 홈 화면입니다.

**표시 정보:**
- 소속 병동명
- 등록된 간호사 수
- 생성된 근무표 수
- 이번 달 근무표 상태 (미생성 / 작성중 / 확정)

**최근 근무표 목록:**
- 최근 5개 근무표의 연월, 상태, 생성일이 표시됩니다.
- **보기** 버튼을 클릭하면 해당 근무표 상세 화면으로 이동합니다.

**빠른 액션 버튼:**
- `근무표 생성` — 근무표 생성 화면으로 이동
- `간호사 관리` — 간호사 관리 화면으로 이동
- `규칙 설정` — 근무 규칙 설정 화면으로 이동

---

### 4.2 간호사 관리

사이드바에서 **간호사 관리**를 클릭합니다.

#### 간호사 목록 조회
- 소속 병동의 전체 간호사 목록이 표시됩니다.
- 각 간호사의 이름, 이메일, 역할, 숙련도가 표시됩니다.

#### 간호사 추가
1. 우측 상단의 **+ 간호사 추가** 버튼을 클릭합니다.
2. 팝업 창에서 다음 정보를 입력합니다.
   - **이름**: 간호사 이름
   - **이메일**: 로그인에 사용할 이메일 (중복 불가)
   - **숙련도**: 주니어 / 시니어 / 책임 중 선택
   - **초기 비밀번호**: 간호사가 처음 로그인할 때 사용할 비밀번호
3. **추가** 버튼을 클릭하면 등록이 완료됩니다.

#### 간호사 수정
1. 목록에서 해당 간호사의 **수정** 버튼을 클릭합니다.
2. 이름, 이메일, 숙련도를 수정할 수 있습니다.
3. **수정** 버튼을 클릭하면 변경이 저장됩니다.

#### 간호사 삭제
1. 목록에서 해당 간호사의 **삭제** 버튼을 클릭합니다.
2. 확인 대화상자에서 **확인**을 클릭하면 삭제됩니다.

> 관리자(수간호사) 계정은 삭제할 수 없습니다.

**숙련도 등급:**

| 등급 | 표시명 | 설명 |
|------|--------|------|
| JUNIOR | 주니어 | 경력 3년 미만 |
| SENIOR | 시니어 | 경력 3년 이상 |
| CHARGE | 책임 | 팀 리더 / 책임 간호사 |

> 숙련도는 자동 생성 시 각 근무조에 다양한 숙련도의 간호사가 골고루 배치되도록 하는 데 사용됩니다.

### 4.3 근무 규칙 설정

사이드바에서 **근무 규칙**을 클릭합니다.

화면은 **좌측 패널(기본 설정)**과 **우측 패널(인원 설정)** 2개 영역으로 구성됩니다.

#### 좌측 패널 — 기본 설정

| 항목 | 설명 | 기본값 |
|------|------|--------|
| 최대 연속 근무일 수 | 간호사가 연속으로 근무할 수 있는 최대 일수 | 5일 |
| 최대 연속 야간근무(N) 수 | 야간근무를 연속으로 할 수 있는 최대 일수 | 3일 |
| 월간 최소 휴무일 수 | 한 달에 보장되어야 하는 최소 휴무일 | 8일 |

#### 좌측 패널 — 근무 전환 규칙

토글 스위치로 특정 근무 전환 패턴의 허용 여부를 설정합니다.

| 토글 | 설명 | 기본값 |
|------|------|--------|
| N→D 허용 | 밤번 다음 날 낮번 허용 | OFF (금지) |
| N→E 허용 | 밤번 다음 날 저녁번 허용 | OFF (금지) |
| NOD 허용 | 밤번→휴무→낮번 패턴 허용 | OFF (금지) |

> 기본적으로 모든 전환 규칙이 **금지**로 설정되어 있습니다. 병동 사정에 따라 허용이 필요하면 토글을 ON으로 변경합니다.

#### 우측 패널 — 근무 인원 설정

**평일 근무 인원:**

| 근무 유형 | 설명 | 기본값 |
|----------|------|--------|
| 낮번(D) 인원 | 평일 주간 근무 최소 인원 | 3명 |
| 저녁번(E) 인원 | 평일 저녁 근무 최소 인원 | 2명 |
| 밤번(N) 인원 | 평일 야간 근무 최소 인원 | 2명 |

**주말/공휴일 근무 인원:**

| 근무 유형 | 설명 | 기본값 |
|----------|------|--------|
| 낮번(D) 인원 | 주말 주간 근무 최소 인원 | 2명 |
| 저녁번(E) 인원 | 주말 저녁 근무 최소 인원 | 2명 |
| 밤번(N) 인원 | 주말 야간 근무 최소 인원 | 2명 |

#### 저장

설정을 변경한 뒤 우측 상단의 **저장** 버튼을 클릭합니다. 저장이 완료되면 버튼 텍스트가 "저장 완료"로 변경됩니다.

> 근무 규칙은 병동별로 독립 관리됩니다. 설정 변경은 다음 번 근무표 자동 생성부터 적용됩니다.

### 4.4 근무표 자동 생성

사이드바에서 **근무표**를 클릭합니다.

#### 생성 절차

1. **대상 연월 선택**: 근무표를 생성할 년/월을 선택합니다.
2. **자동 생성** 버튼을 클릭합니다.
3. 시스템이 SA(Simulated Annealing) 알고리즘을 실행하여 최적의 근무표를 생성합니다.
4. 생성이 완료되면 자동으로 **근무표 상세 화면**으로 이동합니다.

**생성 시 자동으로 반영되는 사항:**
- 근무 규칙 설정에서 지정한 모든 제약 조건
- 승인된 간호사 근무 요청
- 이전 달 근무표 (연속성 보장)

**생성 중 표시:**
- 버튼이 "생성 중..."으로 변경되며, 아래에 "SA 알고리즘으로 최적 근무표를 생성하고 있습니다..." 메시지가 표시됩니다.
- 생성이 완료될 때까지 페이지를 벗어나지 마세요.

**생성 소요 시간:**
- 간호사 10명 기준: 약 10~15초
- 간호사 20명 이상: 약 15~30초

> 같은 연월에 대해 여러 번 생성할 수 있습니다. 기존 작성중(DRAFT) 상태의 근무표는 새로 생성 시 자동으로 대체됩니다.

### 4.5 근무표 조회 및 수정

#### 근무표 테이블 뷰

근무표 상세 화면에서는 모든 간호사의 월간 근무 일정을 한눈에 볼 수 있습니다.

**상단 영역:**
- 좌측: 연월 제목 (예: "2026-02 근무표") + 상태 뱃지 (`작성중` 또는 `확정`)
- 우측: `Excel 내보내기` / `통계` / `확정` 버튼

**테이블 영역:**

```
간호사   1   2   3   4   5   6   7  ...  27  28   D   E   N   O
         일  월  화  수  목  금  토       금  토
강도현   N   N   O   E   E   D   N  ...   E   D    8   6   5   9
김수간   E   O   O   D   D   E   D  ...   O   O    9   5   4  10
박지민   D   D   E   O   O   O   D  ...   E   D    7   7   5   9
...
```

- **행**: 간호사 이름 (좌측 고정 — 스크롤해도 항상 보임)
- **열**: 날짜 (1일~말일), 각 날짜 아래에 요일 표시
- **셀**: 근무 유형 코드가 색상 박스로 표시
- **우측 통계열**: 간호사별 D / E / N / O 횟수 (각 색상으로 표시)
- **주말/공휴일**: 날짜 헤더의 숫자와 요일이 **빨간색**으로 표시
- 테이블이 화면보다 넓으면 **가로 스크롤**이 가능합니다 (간호사 이름 열은 고정)

**색상 코딩:**
- `D` (낮번): 파란색
- `E` (저녁번): 초록색
- `N` (밤번): 보라색
- `O` (휴무): 회색
- `X` (기타): 노란색

#### 개별 셀 수정

> 근무표가 **작성중(DRAFT)** 상태일 때만 수정 가능합니다.

1. 변경하고 싶은 셀을 **클릭**합니다.
2. 근무 유형 선택 팝업이 표시됩니다 (`D` `E` `N` `O` `X`).
3. 원하는 근무 유형을 클릭하면 즉시 변경됩니다.
4. 변경 후 **제약 조건이 자동으로 재검증**됩니다.

**제약 위반 표시:**
- 하드 제약을 위반하는 셀에는 **빨간색 테두리**가 표시됩니다.
- 화면 상단에 "하드 제약 위반 N건이 있습니다" 경고 배너가 표시됩니다.
- 화면 하단의 **제약 조건 검증 결과** 패널에서 모든 위반 사항을 확인할 수 있습니다.
  - 하드 제약 위반: 빨간색 텍스트
  - 소프트 제약 위반: 회색 텍스트 (페널티 점수 표시)

### 4.6 근무표 확정 및 내보내기

#### 근무표 확정

1. 근무표 상세 화면에서 우측 상단의 **확정** 버튼을 클릭합니다.
2. 확인 대화상자에서 **확인**을 클릭합니다.
3. 근무표 상태가 **작성중(DRAFT)** → **확정(CONFIRMED)** 으로 변경됩니다.
4. 확정된 근무표는 더 이상 셀 수정이 불가능합니다.

> 확정 전에 하드 제약 위반이 없는지 반드시 확인하세요.

#### Excel 내보내기

1. 근무표 상세 화면에서 **Excel 내보내기** 버튼을 클릭합니다.
2. `.xlsx` 파일이 자동으로 다운로드됩니다.
3. 파일에는 간호사별 월간 근무 일정과 D/E/N/O/X 횟수 통계가 포함됩니다.

### 4.7 근무 요청 관리

사이드바에서 **근무 요청**을 클릭합니다.

관리자는 간호사들이 제출한 근무 요청 목록을 볼 수 있습니다.

| 열 | 설명 |
|----|------|
| 간호사 | 요청을 제출한 간호사 이름 |
| 날짜 | 요청 대상 날짜 |
| 요청 근무 | 요청한 근무 유형 (D/E/N/O/X) |
| 사유 | 간호사가 입력한 사유 (선택 사항) |
| 상태 | 대기중 / 승인 / 반려 |

#### 승인 / 반려

- **대기중** 상태의 요청에 대해 **승인** 또는 **반려** 버튼이 표시됩니다.
- **승인**: 다음 자동 생성 시 해당 요청이 소프트 제약으로 반영됩니다.
- **반려**: 해당 요청이 무시됩니다.

> 승인된 요청은 자동 생성 시 가능한 한 반영되지만, 다른 제약 조건과 충돌하는 경우 반영되지 않을 수 있습니다.

### 4.8 통계 대시보드

근무표 목록에서 **통계** 버튼을 클릭하거나, 근무표 상세 화면에서 **통계** 버튼을 클릭합니다.

#### 요약 지표 (상단 카드)

| 지표 | 설명 |
|------|------|
| 간호사 수 | 해당 병동의 총 간호사 수 |
| 평균 근무일 | 간호사 1인당 평균 근무일 수 |
| 평균 휴무일 | 간호사 1인당 평균 휴무일 수 |
| 요청 반영률 | 승인된 요청 중 실제 반영된 비율 (%) |

#### 공정성 지표

근무 유형별 **표준편차**를 표시합니다. 표준편차가 낮을수록 간호사 간 배분이 공정합니다.

- `D 편차`: 낮번 배분 균등도
- `E 편차`: 저녁번 배분 균등도
- `N 편차`: 밤번 배분 균등도

#### 간호사별 상세 통계 (테이블)

| 열 | 설명 |
|----|------|
| 이름 | 간호사 이름 |
| D / E / N / O | 각 근무 유형별 횟수 |
| 총 근무 | D + E + N 합계 |
| 주말 근무 | 주말/공휴일 근무 횟수 |
| 요청 반영 | 반영된 요청 / 전체 요청 수 |

#### 근무 분포 (바 차트)

간호사별로 D/E/N/O의 비율을 가로 바 차트로 시각화합니다. 색상은 근무 유형별 코딩과 동일합니다.

## 5. 간호사 기능

### 5.1 내 근무 일정 조회

사이드바에서 **근무표**를 클릭합니다.

- 근무표 목록에서 조회하고 싶은 연월의 **보기** 버튼을 클릭합니다.
- 전체 간호사의 근무표가 표시되며, 자신의 일정을 확인할 수 있습니다.
- 동료의 근무 일정도 같은 화면에서 확인 가능합니다.

> 간호사 계정으로는 셀 수정이 불가능합니다 (읽기 전용).

### 5.2 근무 요청 제출

사이드바에서 **근무 요청**을 클릭합니다.

#### 요청 제출

1. **+ 요청 제출** 버튼을 클릭합니다.
2. 팝업 창에서 다음 정보를 입력합니다.
   - **대상 근무표**: 요청을 적용할 근무표 선택
   - **날짜**: 요청 대상 날짜
   - **요청 근무 유형**: 원하는 근무 유형 선택 (D/E/N/O/X)
   - **사유**: 요청 사유 (선택 입력)
3. **제출** 버튼을 클릭합니다.

**제한사항:**
- **월 최대 2건**까지만 요청할 수 있습니다.
- 2건을 초과하면 "월 최대 2건의 요청만 가능합니다" 오류가 표시됩니다.
- 제출된 요청은 관리자(수간호사)가 승인/반려합니다.

#### 내 요청 목록

- 자신이 제출한 요청의 날짜, 요청 근무, 사유, 상태를 확인할 수 있습니다.
- 상태: **대기중** (노란색) / **승인** (초록색) / **반려** (빨간색)

## 7. 제약 조건 안내

자동 생성 알고리즘(SA)이 적용하는 제약 조건입니다.

### 하드 제약 (반드시 충족)

위반 시 근무표가 유효하지 않습니다. 빨간색 경고로 표시됩니다.

| ID | 제약 | 설명 |
|----|------|------|
| HC-1 | 최소 인원 충족 | 매일 각 근무 유형(D/E/N)에 설정된 최소 인원이 배정되어야 합니다 |
| HC-2 | 연속 근무 제한 | 설정된 최대 연속 근무일을 초과할 수 없습니다 (기본 5일) |
| HC-3 | 금지 전환 패턴 | N→D, N→E, NOD 등 설정된 금지 패턴이 존재하면 안 됩니다 |
| HC-4 | 나이트 보호 | 나이트 단독 근무(1일만 N)와 4일 이상 연속 나이트가 금지됩니다 |

### 소프트 제약 (가능한 한 충족)

위반 시 페널티 점수가 부과되며, 알고리즘이 최소화하려고 합니다.

| ID | 제약 | 설명 |
|----|------|------|
| SC-1 | 요청 반영 | 승인된 간호사 근무 요청을 가능한 한 반영합니다 |
| SC-2 | 균등 배분 | 간호사 간 D/E/N 횟수, 주말 근무 횟수를 균등하게 배분합니다 |
| SC-2b | 숙련도 믹스 | 각 근무조에 다양한 숙련도의 간호사를 골고루 배치합니다 |
| SC-3 | 연속 휴무 | 휴무일이 연속으로 배치되면 보너스 점수를 부여합니다 |

## 8. 자주 묻는 질문 (FAQ)

### Q. 근무표 생성 시 "병동 또는 규칙이 존재하지 않습니다" 오류가 뜹니다.
**A.** 병동에 근무 규칙이 설정되어 있지 않을 수 있습니다. **근무 규칙** 메뉴에서 규칙을 확인하고 저장해 주세요.

### Q. 자동 생성했는데 빨간색 위반 표시가 있습니다.
**A.** 간호사 수가 최소 인원 요구사항에 비해 부족할 수 있습니다. 예를 들어, 평일 D=3, E=2, N=2이면 최소 7명이 매일 근무해야 하므로 휴무를 고려하면 최소 9~10명의 간호사가 필요합니다. 간호사를 추가하거나 필요 인원 수를 줄여보세요.

### Q. 특정 간호사의 요청이 반영되지 않았습니다.
**A.** 근무 요청은 소프트 제약으로 반영됩니다. 다른 하드 제약(최소 인원, 금지 패턴 등)과 충돌하는 경우 반영되지 않을 수 있습니다. 이는 정상적인 동작입니다.

### Q. N→D 전환을 허용하고 싶습니다.
**A.** **근무 규칙** 화면에서 "N→D 허용" 토글을 ON으로 변경한 뒤 **저장**을 클릭하세요. 다음 자동 생성부터 적용됩니다.

### Q. 확정한 근무표를 다시 수정하고 싶습니다.
**A.** 확정(CONFIRMED) 상태의 근무표는 직접 수정할 수 없습니다. 같은 연월로 새로 자동 생성하면 기존 작성중(DRAFT) 근무표를 대체할 수 있습니다.

### Q. Excel로 내보낸 파일에 어떤 정보가 포함되나요?
**A.** 간호사 이름, 1일~말일 근무 유형, D/E/N/O/X 횟수 통계가 포함됩니다. 파일명은 `schedule_YYYY-MM.xlsx` 형식입니다.

### Q. 간호사 계정의 비밀번호를 변경하고 싶습니다.
**A.** 현재 버전에서는 비밀번호 변경 UI가 제공되지 않습니다. 관리자가 간호사를 삭제 후 새 비밀번호로 재등록하거나, 데이터베이스에서 직접 변경해야 합니다.

### Q. http://localhost:5173 접속 시 "연결을 거부했습니다" 오류가 뜹니다.
**A.** 프론트엔드 서버(Vite)가 실행되지 않은 상태입니다. `npm run dev:backend`만 실행하면 백엔드(포트 3000)만 기동됩니다. **프로젝트 루트에서 `npm run dev`를 실행**하면 백엔드와 프론트엔드가 동시에 시작됩니다. 프론트엔드만 별도로 실행하려면 `npm run dev:frontend`를 사용하세요.

### Q. http://localhost:3000 접속 시 "Cannot GET /" 오류가 뜹니다.
**A.** 정상 동작입니다. 백엔드 서버는 API 전용 서버로 `/api/*` 경로만 제공하며, 루트(`/`) 경로에는 페이지가 없습니다. **브라우저에서는 반드시 http://localhost:5173 으로 접속**하세요. 프론트엔드가 API 호출을 자동으로 백엔드(포트 3000)로 프록시합니다.
