## 1. 프로젝트 개요

### 1.1 서비스명
- **NurseSch** (Nurse Scheduler)

### 1.2 목적
병원 간호사를 위한 월간 근무 일정표(근무표)를 **자동으로 생성**해 주는 웹서비스.
하드/소프트 제약 조건을 모두 반영하여 공정하고 효율적인 근무표를 생성하며,
수작업 대비 시간을 대폭 단축한다 (목표: 5초 이내 생성).

### 1.3 타겟 사용자
| 사용자 | 설명 |
|--------|------|
| **HN (수간호사 / Head Nurse)** | 근무표 생성, 규칙 설정, 병동/간호사 관리, 결과 수정 및 확정 |
| **CN (책임간호사 / Charge Nurse)** | 개인 근무 일정 조회, 선호도/휴무 요청 제출, 동료 일정 확인 |
| **RN (평간호사 / Registered Nurse)** | 개인 근무 일정 조회, 선호도/휴무 요청 제출 |
| **AN (보조간호사 / Assistant Nurse)** | 개인 근무 일정 조회 (Mid근무 기본 배정) |

## 2. 사용자 역할 정의

### 2.1 관리자 (HN — Head Nurse / 수간호사)
- 병동 및 소속 간호사 등록/관리
- 근무 규칙(제약 조건) 설정 및 커스터마이징
- 자동 근무표 생성 실행
- 생성된 근무표 조회, 수동 수정, 확정/배포
- 통계 및 공정성 대시보드 조회
- 간호사 근무 요청 승인/반려
- 간호사별 희망 휴일 직접 지정
- 간호사별 연차(Y) 수 설정

### 2.2 일반 간호사 (CN / RN / AN)
- 개인 근무 일정 조회 (월별/주별)
- 근무 선호도 및 휴무 요청 제출 (월 최대 2건)
- 동료 근무 일정 조회 (같은 병동 내)
- 근무 유형별 개인 통계 조회

## 3. 시스템 아키텍처 요구사항

### 3.1 레이어 구조 (CLAUDE.md 기반)

```
┌─────────────────────────────────────────┐
│            UI Layer                     │
│  근무표 입력, 결과 표시, 대시보드        │
├─────────────────────────────────────────┤
│         Scheduling Engine               │
│  SA 기반 자동 배정 알고리즘              │
├─────────────────────────────────────────┤
│       Constraint Validator              │
│  하드/소프트 제약 검증 (엔진과 분리)      │
├─────────────────────────────────────────┤
│       Persistence Layer                 │
│  근무표, 간호사 프로필, 설정 저장         │
└─────────────────────────────────────────┘
```

### 3.2 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| **프론트엔드** | React + TypeScript + Vite | SPA, 반응형 웹 |
| **백엔드** | Python 3.11+ + FastAPI | REST API 서버 |
| **스케줄링 엔진** | Python | SA 알고리즘 구현 |
| **ORM** | SQLAlchemy + Alembic | 마이그레이션 포함 |
| **데이터베이스** | SQLite | 개발/소규모 배포 (PostgreSQL 확장 가능) |
| **인증** | JWT (python-jose) | 역할 기반 접근 제어 (RBAC) |
| **Excel 내보내기** | openpyxl | 근무표 Excel 출력 |

### 3.3 핵심 설계 원칙
- **Scheduling Engine과 Constraint Validator를 분리**: 제약 조건 추가/변경 시 엔진 코드 수정 불필요
- **병동별 제약 조건 커스터마이징**: 각 병동마다 고유한 규칙과 가중치 설정 가능
- **API 우선 설계**: 향후 모바일 앱 확장 대비
- **UI/UX 스타일**: 최신 트렌드 반영, 미니멀한 디자인

## 4. 도메인 모델 정의

### 4.1 근무 유형 (Shift Types)

| 코드 | 명칭 | 설명 | UI 색상 |
|------|------|------|---------|
| `D` | 데이 (Day) | 주간 근무 | 파란색 (#4A90D9) |
| `E` | 이브닝 (Evening) | 오후~저녁 근무 | 초록색 (#7BC67E) |
| `N` | 나이트 (Night) | 야간 근무 | 보라색 (#9B59B6) |
| `M` | 미드 (Mid) | 09:00~17:00, AN 기본 근무 | 하늘색 (#5DADE2) |
| `O` | 오프 (Off) | 당일 비근무 | 회색 (#BDC3C7) |
| `Y` | 연차 (Annual Leave) | 연가 사용 | 주황색 (#E67E22) |
| `X` | 기타 | 교육, 출장 등 | 노란색 (#F1C40F) |

### 4.2 간호사 직급 (Nurse Grade)

> 기존 skill_level(JUNIOR/SENIOR/CHARGE)을 직급(grade) 체계로 변경

| 코드 | 명칭 | 설명 | 시스템 역할 |
|------|------|------|-----------|
| `HN` | Head Nurse (수간호사) | 병동 관리자 | ADMIN |
| `CN` | Charge Nurse (책임간호사) | Charge 담당 가능 | NURSE |
| `RN` | Registered Nurse (평간호사) | 일반 근무 담당 | NURSE |
| `AN` | Assistant Nurse (보조간호사) | Mid근무 기본, 보조 역할 | NURSE |

### 4.3 핵심 엔티티

- **간호사 (Nurse)**: 이름, ID, 소속 병동, 직급(grade), 정렬 순서(sort_order), 나이트 전담 여부, 월 연차 수
- **병동 (Ward)**: 병동명, 관리자(HN), 소속 간호사 목록
- **근무표 (Schedule)**: 대상 병동, 대상 연월, 상태(작성중/확정), 셀 데이터
- **근무 규칙 (Rule)**: 병동별 제약 조건 설정값 (Charge/Action 인원, 나이트 전담 설정 포함)
- **근무 요청 (ShiftRequest)**: 간호사별 특정 날짜/근무 유형 요청 (월 최대 2건)
- **연차 설정 (NurseMonthlyLeave)**: 간호사별 월별 연차 수, 관리자 지정 희망 휴일

## 5. 기능 요구사항 (Functional Requirements)

### FR-1: 병동/간호사 관리

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-1.1 | 관리자가 병동을 생성/수정/삭제할 수 있다 | 높음 |
| FR-1.2 | 관리자가 병동에 간호사를 등록/수정/삭제할 수 있다 | 높음 |
| FR-1.3 | 간호사 정보에 이름, 이메일, 직급(HN/CN/RN/AN)을 포함한다 | 높음 |
| FR-1.4 | 간호사 목록을 이름/이메일/직급 기준으로 정렬할 수 있다 | 높음 |
| FR-1.5 | 관리자가 간호사 목록의 순서를 직접 지정할 수 있다 (drag & drop 또는 순서 번호 직접 입력). 지정된 순서는 모든 정렬보다 우선순위가 높다 | 높음 |
| FR-1.6 | 간호사 수정 팝업이 키보드 입력 중 닫히지 않아야 한다 (이벤트 버블링 차단) | 높음 |

### FR-2: 근무표 기본 설정

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-2.1 | 관리자가 스케줄 대상 연월을 선택할 수 있다 | 높음 |
| FR-2.2 | 평일/주말별 근무 유형별 필요 Charge 인원 수와 Action 인원 수를 각각 설정할 수 있다 | 높음 |
| FR-2.3 | 공휴일을 별도로 지정할 수 있다 (주말과 동일 인원 적용) | 중간 |
| FR-2.4 | 이전 달 근무표를 참조하여 연속성을 유지할 수 있다 | 높음 |

**Charge/Action 인원 구분:**
- **Charge**: HN 또는 CN이 담당. 각 근무조의 책임자
- **Action**: RN이 담당. 일반 근무 인력
- AN은 Mid근무로 별도 운영 (D/E 근무 지정 가능)

### FR-3: 근무 규칙 설정

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-3.1 | 최대 연속 근무일 수를 설정할 수 있다 (기본값: 5일) | 높음 |
| FR-3.2 | 최대 연속 나이트 수를 설정할 수 있다 (기본값: 3일) | 높음 |
| FR-3.3 | 근무 전환 규칙(금지 패턴)을 설정할 수 있다 | 높음 |
| FR-3.4 | 월간 최소 휴무일 수를 설정할 수 있다 | 중간 |
| FR-3.5 | 병동별로 독립적인 규칙 설정이 가능하다 | 높음 |
| FR-3.6 | AN은 특별히 지정하지 않으면 주말(토/일)을 자동 Off 처리한다 | 높음 |
| FR-3.7 | 나이트 전담 간호사를 지정할 수 있다. 지정된 전담 간호사는 나이트 근무만 배정된다 | 높음 |
| FR-3.8 | 나이트 전담 간호사가 등록된 경우, 일반 간호사(비전담)는 나이트 근무에 자동 배정되지 않는다 | 높음 |
| FR-3.9 | AN은 기본적으로 Mid(09:00~17:00) 근무로 배정되며, 필요 시 Day 또는 Evening으로 지정할 수 있다 | 높음 |

**근무 전환 금지 패턴 (기본값):**
| 패턴 | 설명 | 기본 설정 |
|------|------|----------|
| N→D | 나이트 후 다음 날 데이 | 금지 |
| N→E | 나이트 후 다음 날 이브닝 | 금지 |
| NOD | 나이트→오프→데이 연속 | 금지 |

### FR-4: 제약 조건 관리

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-4.1 | 하드 제약과 소프트 제약을 구분하여 관리한다 | 높음 |
| FR-4.2 | 각 제약 조건별 가중치(페널티 점수)를 병동별로 설정할 수 있다 | 높음 |
| FR-4.3 | 제약 조건 위반 시 실시간으로 피드백을 제공한다 | 중간 |

### FR-5: 자동 근무표 생성

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-5.1 | Simulated Annealing 알고리즘을 사용하여 근무표를 자동 생성한다 | 높음 |
| FR-5.2 | 생성 시 모든 하드 제약을 100% 충족한다 | 높음 |
| FR-5.3 | 소프트 제약은 가중치 기반 페널티 시스템으로 최적화한다 | 높음 |
| FR-5.4 | 이전 달 근무표를 참조하여 연속성을 보장한다 | 높음 |
| FR-5.5 | 나이트 전담 간호사는 나이트 근무에만 배정된다 | 높음 |
| FR-5.6 | AN은 기본 Mid 근무로 배정되고, 주말/공휴일 자동 Off 규칙을 적용한다 | 높음 |
| FR-5.7 | 각 간호사의 총 비근무일수(OFF+Y)를 근무표에 반영한다 | 높음 |
| FR-5.8 | 희망 근무 요청 충돌로 자동 생성이 불가한 경우 충돌 내역을 표시한다 | 높음 |
| FR-5.9 | 생성된 근무표를 Excel로 내보낼 수 있다 | 높음 |

### FR-6: 근무표 조회/수정

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-6.1 | 생성된 근무표를 테이블 형태로 조회한다 (행: 간호사, 열: 날짜) | 높음 |
| FR-6.2 | 관리자가 개별 셀(특정 간호사의 특정 날짜 근무)을 수동 수정할 수 있다 | 높음 |
| FR-6.3 | 수정 시 제약 조건 위반 여부를 실시간으로 검증하고 경고를 표시한다 | 높음 |
| FR-6.4 | 근무표를 확정/배포할 수 있다 | 높음 |
| FR-6.5 | 근무 요청 충돌로 자동 생성이 불가한 경우, 관리자가 충돌 요청을 직접 수정하여 재생성할 수 있다 | 높음 |

### FR-7: 개인 선호도/근무 요청 및 연차 관리

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FR-7.1 | 간호사가 특정 날짜에 대한 근무/휴무 요청을 제출할 수 있다 (월 최대 2건) | 높음 |
| FR-7.2 | 관리자가 요청 목록을 조회하고 승인/반려할 수 있다 | 높음 |
| FR-7.3 | 승인된 요청은 자동 생성 시 소프트 제약으로 반영된다 | 높음 |
| FR-7.4 | 근무표 작성 월에 간호사별로 지정 휴일수와 희망 휴일을 표시한다 | 높음 |
| FR-7.5 | 관리자(HN)가 각 간호사별 희망 휴일을 직접 지정/수정할 수 있다 | 높음 |
| FR-7.6 | 간호사별 총 휴일수 = 해당월 주말일수 + 공휴일수 + 개인 연차(Y)수로 자동 계산한다 | 높음 |
| FR-7.7 | 개인 연차는 기본 월 1일이며, 관리자가 간호사별로 변경/지정할 수 있다 | 높음 |
| FR-7.8 | 계산된 총 비근무일수(OFF + Y)를 근무표 생성 시 반영한다 | 높음 |

### FR-8: 통계 및 공정성 대시보드

| FR-8.3 | 주말/공휴일 근무 횟수 균등 여부를 시각화한다 | 중간 |

## 7. 알고리즘 설계 요구사항

> 간호사_근무표_생성을_위한_알고리즘.pdf 기반

### 7.1 알고리즘 선택: Simulated Annealing (SA)

**선택 이유:**
- Greedy 알고리즘은 2~3개 제약 조건만으로도 지역 최적해(local optimum)에 빠짐
- SA는 확률적 수락을 통해 지역 최적해를 탈출하고 전역 최적해에 근접 가능
- 다수의 상충하는 제약 조건을 동시에 처리할 수 있음

### 7.2 SA 파라미터

| 파라미터 | 설명 | 기본값 |
|---------|------|--------|
| `INITIAL_TEMPERATURE` | 초기 온도 | 1000 |
| `COOLING_RATE` | 냉각률 | 0.9993 |
| `MAX_ITERATIONS` | 최대 반복 횟수 | 15000 |
| `MAX_NO_IMPROVEMENT` | 개선 없는 최대 반복 | 1500 |

**재가열 메커니즘 (Reheat):**
- `MAX_NO_IMPROVEMENT` 횟수 동안 개선이 없으면 온도를 초기값으로 리셋

### 7.3 직급별 배정 전략

| 직급 | 나이트 전담 아닐 때 | 나이트 전담일 때 |
|------|-----------------|--------------|
| HN | Charge 역할 (D/E/N/O) | (해당 없음) |
| CN | Charge 역할 (D/E/N/O) | (해당 없음) |
| RN | Action 역할 (D/E/N/O) | (해당 없음) |
| AN | Mid 기본 (D/E 가능, 주말 자동 Off) | (해당 없음) |
| 나이트 전담 | N만 배정 | N만 배정 |

### 7.4 초기 해(Initial Solution) 생성
1. 나이트 전담 간호사: N 근무 우선 배정
2. AN: Mid 우선 배정, 주말/공휴일 Off 처리
3. 관리자가 지정한 희망 휴일 선적용
4. 승인된 근무 요청 선적용
5. Charge 인원(HN/CN) 먼저 채우고, Action 인원(RN) 채움
6. 각 간호사의 총 비근무일수 보장

### 7.5 평가 함수 구조

```
총 점수 = Σ (제약별 평가 함수 × 가중치)
```

| 평가 함수 | 설명 |
|----------|------|
| `evaluateShiftRequirements` | 일별 D/E/N Charge+Action 인원 충족 여부 |
| `evaluateConsecutiveShifts` | 연속 근무일, 연속 나이트 초과 검사 |
| `evaluateWorkloadBalance` | 간호사 간 근무 유형 횟수 편차 최소화 |
| `evaluateShiftRequests` | 승인된 요청의 반영 여부 |
| `evaluateForbiddenPatterns` | N→D, N→E, NOD 패턴 |
| `evaluateOffDayClustering` | 연속 휴무 보너스 점수 |
| `evaluateTotalOffDays` | 간호사별 총 비근무일수(OFF+Y) 목표 달성 |

## 8. 제약 조건 상세 명세

### 8.1 하드 제약 (Hard Constraints)

| ID | 규칙 | 설명 |
|----|------|------|
| HC-1 | Charge/Action 인원 충족 | 매일 각 근무조(D/E/N)에 Charge(HN/CN) 1명 이상 + Action(RN) 최소 인원 배정 |
| HC-2 | 최대 연속 근무일 제한 | 연속 근무일이 설정값(기본 5일) 초과 금지 |
| HC-3 | 금지 전환 패턴 | N→D, N→E, NOD 패턴 금지 (개별 토글) |
| HC-4 | 나이트 근무 보호 | 나이트 단독(1일) 금지, 연속 4일 이상 나이트 금지 |
| HC-5 | 나이트 전담 준수 | 나이트 전담 지정자는 N 근무만 배정 |
| HC-6 | AN 주말 Off | AN은 주말(토/일) 자동 Off (별도 지정 시 예외) |

### 8.2 소프트 제약 (Soft Constraints)

| ID | 규칙 | 설명 | 가중치 |
|----|------|------|--------|
| SC-1 | 개인 근무 요청 반영 | 승인된 요청(월 최대 2건)을 반영 | 중간 |
| SC-2 | 균등한 업무 유형 배분 | D/E/N/O 횟수 편차 최소화 | 높음 |
| SC-2a | 주말 근무 균등 | 주말/공휴일 근무 횟수 균등 분배 | 높음 |
| SC-2b | 직급별 조 구성 | 각 근무조에 직급 다양성 보장 | 중간 |
| SC-3 | 휴무일 연속 배치 | 연속 휴무 시 보너스 | 낮음 |
| SC-4 | 총 비근무일수 달성 | 각 간호사의 목표 비근무일수 달성 | 높음 |
| SC-5 | 관리자 지정 희망 휴일 반영 | HN이 직접 지정한 휴일 반영 | 높음 |

### 8.3 가중치 체계
```
하드 제약 위반: 1000점 이상 (사실상 무효화)
소프트 제약 (높음): 50~100점
소프트 제약 (중간): 20~50점
소프트 제약 (낮음): 5~20점
```

## 9. UI/UX 설계 요구사항

### 9.1 전체 화면 구성

| 화면 | 사용자 | 설명 |
|------|--------|------|
| 로그인 | 공통 | 이메일/비밀번호 인증 |
| 대시보드 | HN | 병동 현황, 근무표 상태 요약 |
| 간호사 관리 | HN | 간호사 등록/수정/삭제, 직급 설정, 순서 관리 |
| 근무 규칙 설정 | HN | 제약 조건, 전환 규칙 토글, Charge/Action 인원, 나이트 전담 설정 |
| 근무표 생성/조회/수정 | HN | 자동 생성, 테이블 뷰, 셀 수정, 확정 |
| 통계 대시보드 | HN | 공정성 지표, 근무 분포 |
| 근무 요청/연차 관리 | HN+간호사 | 요청 제출, 승인/반려, 희망 휴일 설정, 연차 관리 |

### 9.2 간호사 관리 화면 상세

- **목록 컬럼**: 순서번호, 이름, 이메일, 직급, 나이트전담 여부, 월연차수, 수정/삭제
- **정렬 기능**: 이름/이메일/직급별 클릭 정렬 (지정 순서 > 이름 > 이메일 > 직급 우선순위)
- **순서 지정**: 드래그 앤 드롭 또는 순서 번호 직접 입력으로 관리자 지정 순서 설정
- **수정 팝업**: 모달 내부 클릭/키보드 입력 시 이벤트 버블링을 차단하여 팝업이 닫히지 않도록 구현

### 9.3 근무 규칙 설정 화면 상세

**좌측 패널:**
- 스케줄 대상 연월 선택
- 근무 전환 규칙 토글 (N→D, N→E, NOD)
- AN 주말 자동 Off 토글 (기본: ON)

**우측 패널:**
- 최대 연속 근무일 수
- 최대 연속 나이트 수
- 월간 최소 휴무일 수
- 평일/주말별 Charge 인원(HN/CN) 수 및 Action 인원(RN) 수
- 나이트 전담 간호사 지정 (멀티셀렉트)

### 9.4 근무표 화면 상세

- **셀 색상**: D=파란, E=초록, N=보라, M=하늘, O=회색, Y=주황, X=노란
- **직접 수정**: 관리자 클릭 시 드롭다운으로 근무 유형 변경, 변경 즉시 위반 검증
- **충돌 표시**: 희망 근무 요청 충돌 시 해당 셀 강조 + 상세 내역 패널 표시
- **수정 이력**: 수동 수정된 셀은 별도 표식

### 9.5 근무 요청/연차 관리 화면 상세

- **간호사별 탭**: 월별로 각 간호사의 총 휴일수(주말+공휴일+연차) 자동 계산 표시
- **희망 휴일 달력**: 달력 UI로 희망 휴일 날짜 선택 (간호사 본인 또는 HN 지정)
- **연차 수 설정**: HN이 간호사별 월 연차 수를 직접 입력
- **총 비근무일수 표시**: `주말일수 + 공휴일수 + 연차수 = 목표 비근무일수`

```
Ward (병동)
├── id (PK)
├── name
└── timestamps

Nurse (간호사)
├── id (PK)
├── ward_id (FK → Ward)
├── name
├── email (UNIQUE)
├── grade (HN | CN | RN | AN)  ← skill_level 대체
├── sort_order (INT, 관리자 지정 순서)
├── is_night_dedicated (BOOL, 나이트 전담)
├── monthly_annual_leave (INT, 월 기본 연차수, default=1)
├── password_hash
└── timestamps

Rule (근무 규칙)
├── id (PK)
├── ward_id (FK → Ward, UNIQUE)
├── max_consecutive_work_days (default: 5)
├── max_consecutive_night_shifts (default: 3)
├── min_monthly_off_days
├── weekday_charge_count (default: 1)    ← Charge(HN/CN) 인원
├── weekday_action_count (default: 2)    ← Action(RN) 인원
├── weekday_night_charge_count (default: 1)
├── weekday_night_action_count (default: 1)
├── weekend_charge_count (default: 1)
├── weekend_action_count (default: 1)
├── weekend_night_charge_count (default: 1)
├── weekend_night_action_count (default: 1)
├── allow_night_to_day (default: false)
├── allow_night_to_evening (default: false)
├── allow_night_off_day (default: false)
├── an_auto_weekend_off (default: true)  ← AN 주말 자동 Off
├── constraint_weights (JSON)
└── timestamps

Schedule (근무표)
├── id (PK)
├── ward_id (FK → Ward)
├── year_month (YYYY-MM)
├── status (DRAFT | CONFIRMED)
└── timestamps

ScheduleEntry (근무표 셀)
├── id (PK)
├── schedule_id (FK → Schedule)
├── nurse_id (FK → Nurse)
├── date
├── shift_type (D | E | N | M | O | Y | X)
└── is_manually_edited (BOOL)

ShiftRequest (근무 요청)
├── id (PK)
├── nurse_id (FK → Nurse)
├── schedule_id (FK → Schedule)
├── date
├── requested_shift_type (D | E | N | M | O | Y | X)

NurseMonthlyLeave (간호사 월별 연차/희망 휴일)

├── annual_leave_count (INT, 해당월 연차수)
├── requested_off_dates (JSON, 희망 휴일 날짜 목록)
└── timestamps

Holiday (공휴일/특수일)
├── id (PK)
├── ward_id (FK → Ward)
├── date
├── name
└── type (HOLIDAY | EDUCATION | OTHER)
```
