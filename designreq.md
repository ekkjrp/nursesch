# 상세 개발 요구사항 (Design Requirements)

> 본 문서는 간호사 근무표 자동 생성 웹서비스의 상세 설계 요구사항을 정의한다.
> CLAUDE.md, references.md, 간호사_근무표_생성을_위한_알고리즘.pdf를 종합하여 작성하였으며,
> 이후 requirements.md와 실제 코드 구현의 기반이 된다.
>
> **작성일**: 2026-02-16

---

## 1. 프로젝트 개요

### 1.1 서비스명
- **NurseSch** (Nurse Scheduler)

### 1.2 목적
병원 간호사를 위한 월간 근무 일정표(근무표)를 **자동으로 생성**해 주는 웹서비스.
하드/소프트 제약 조건을 모두 반영하여 공정하고 효율적인 근무표를 생성하며,
수작업 대비 시간을 대폭 단축한다 (목표: 5초 이내 생성).

### 1.3 타겟 사용자
| 사용자 | 설명 |
|--------|------|
| **관리자 (수간호사)** | 근무표 생성, 규칙 설정, 병동/간호사 관리, 결과 수정 및 확정 |
| **일반 간호사** | 개인 근무 일정 조회, 선호도/휴무 요청 제출, 동료 일정 확인 |

---

## 2. 사용자 역할 정의

### 2.1 관리자 (수간호사 / Head Nurse)
- 병동 및 소속 간호사 등록/관리
- 근무 규칙(제약 조건) 설정 및 커스터마이징
- 자동 근무표 생성 실행
- 생성된 근무표 조회, 수동 수정, 확정/배포
- 통계 및 공정성 대시보드 조회
- 간호사 근무 요청 승인/반려

### 2.2 일반 간호사 (Nurse)
- 개인 근무 일정 조회 (월별/주별)
- 근무 선호도 및 휴무 요청 제출 (월 최대 2건)
- 동료 근무 일정 조회 (같은 병동 내)
- 개인 일정(외출, 교육 등) 등록 및 근무 일정과 병행 관리
- 근무 유형별 개인 통계 조회

## 3. 시스템 아키텍처 요구사항

### 3.1 레이어 구조 (CLAUDE.md 기반)

```
┌─────────────────────────────────────────┐
│            UI Layer                     │
│  근무표 입력, 결과 표시, 대시보드        │
├─────────────────────────────────────────┤
│         Scheduling Engine               │
│  SA 기반 자동 배정 알고리즘              │
├─────────────────────────────────────────┤
│       Constraint Validator              │
│  하드/소프트 제약 검증 (엔진과 분리)      │
├─────────────────────────────────────────┤
│       Persistence Layer                 │
│  근무표, 간호사 프로필, 설정 저장         │
└─────────────────────────────────────────┘
```

### 3.2 기술 스택 제안

| 영역 | 기술 | 비고 |
|------|------|------|
| **프론트엔드** | React + TypeScript | SPA, 반응형 웹 (모바일 대응) |
| **백엔드** | Node.js (Express) 또는 Python (FastAPI) | REST API 서버 |
| **스케줄링 엔진** | Python 또는 TypeScript | SA 알고리즘 구현 |
| **데이터베이스** | PostgreSQL | 관계형 데이터 모델에 적합 |
| **인증** | JWT 기반 토큰 인증 | 역할 기반 접근 제어 (RBAC) |
| **배포** | Docker + Cloud (AWS/GCP) | 확장성 고려 |

### 3.3 핵심 설계 원칙
- **Scheduling Engine과 Constraint Validator를 분리**: 제약 조건 추가/변경 시 엔진 코드 수정 불필요
- **병동별 제약 조건 커스터마이징**: 각 병동마다 고유한 규칙과 가중치 설정 가능
- **API 우선 설계**: 향후 모바일 앱 확장 대비
- **UI/UX스타일**: UI/UX는 최신 트랜드를 반영하고 또한 너무 복잡하지 않게 미니멀하게 구현

## 4. 도메인 모델 정의

### 4.2 핵심 엔티티

- **간호사 (Nurse)**: 이름, ID, 소속 병동, 역할, 숙련도 등급

- **근무 요청 (ShiftRequest)**: 간호사별 특정 날짜/근무 유형 요청 (월 최대 2건)

## 7. 알고리즘 설계 요구사항

> 간호사_근무표_생성을_위한_알고리즘.pdf 기반

### 7.1 알고리즘 선택: Simulated Annealing (SA)

**선택 이유:**
- Greedy 알고리즘은 2~3개 제약 조건만으로도 지역 최적해(local optimum)에 빠짐
- SA는 확률적 수락을 통해 지역 최적해를 탈출하고 전역 최적해에 근접 가능
- 다수의 상충하는 제약 조건을 동시에 처리할 수 있음

### 7.2 SA 파라미터

| 파라미터 | 설명 | 비고 |
|---------|------|------|
| `INITIAL_TEMPERATURE` | 초기 온도 | 높을수록 초기 탐색 범위 넓음 |
| `COOLING_RATE` | 냉각률 (0 < r < 1) | 온도 감소 속도 제어 |
| `MAX_ITERATIONS` | 최대 반복 횟수 | 성능과 품질 사이 트레이드오프 |
| `MAX_NO_IMPROVEMENT` | 개선 없는 최대 반복 | 초과 시 온도 리셋(재가열) |

**재가열 메커니즘 (Reheat):**
- `MAX_NO_IMPROVEMENT` 횟수 동안 개선이 없으면 온도를 `INITIAL_TEMPERATURE`로 리셋
- 지역 최적해 탈출을 위한 핵심 전략

### 7.3 3단계 탐색 과정

| 단계 | 온도 | 변화 크기 | 나쁜 해 수락 확률 | 목적 |
|------|------|---------|-----------------|------|
| 초기 (Phase 1) | 높음 | 큼 | 높음 | 해 공간 넓은 탐색 |
| 중기 (Phase 2) | 중간 | 중간 | 중간 | 지역 최적해 탈출 |
| 후기 (Phase 3) | 낮음 | 작음 | 거의 0 | 최적해 수렴 |

### 7.4 초기 해(Initial Solution) 생성
- 이전 달 근무표(`prevMonthSchedules`)를 기반으로 초기 해를 구성
- 빈 상태가 아닌 기존 스케줄에서 출발하여 연속성을 보장
- 초기 해에 대해 평가 함수를 실행하여 기준 점수(baseline score) 산출

### 7.5 이웃 해(Neighbor Solution) 생성 전략
- 현재 해를 **무작위로 변형**하되, **하드 제약 위반 케이스를 해결하는 방향**으로 변형
- 5가지 금지 패턴(N→D, N→E, NOD, 나이트 단독, 연속 나이트 4일 이상) 중 하나를 해소하는 변형 수행
- 위반 대상 지향적(violation-targeted) 생성으로 탐색 효율 향상

### 7.6 평가 함수 (Objective Function) 구조

```
총 점수 = Σ (제약별 평가 함수 × 병동별 가중치)
```

| 평가 함수 | 대상 제약 | 설명 |
|----------|---------|------|
| `evaluateShiftRequirements` | 근무 유형별 최소 인원 | 일별 D/E/N 인원 충족 여부 |
| `evaluateConsecutiveShifts` | 연속 근무/야간 제한 | 연속 근무일, 연속 야간 초과 검사 |
| `evaluateWorkloadBalance` | 업무량 균등 배분 | 간호사 간 근무 유형 횟수 편차 |
| `evaluateShiftRequests` | 개인 근무 요청 반영 | 승인된 요청의 반영 여부 |
| `evaluateForbiddenPatterns` | 금지 전환 패턴 | N→D, N→E, NOD 패턴 존재 여부 |
| `evaluateOffDayClustering` | 휴무일 연속 배치 | 연속 휴무 보너스 점수 |

### 7.7 해 수락 기준

```
if (neighborScore < currentScore):
    항상 수락 (개선된 해)
else:
    수락 확률 = exp(-delta / temperature)
    난수 < 수락확률 이면 수락 (악화되어도 확률적 수락)
```

- `delta` = neighborScore - currentScore (점수 차이)
- 온도가 낮아질수록 악화된 해의 수락 확률이 0에 수렴

### 7.8 SA 메인 루프 의사코드

```
currentSolution = createInitialSolution(wardSchedule, rule, wardMembers, yearMonth)
bestSolution = currentSolution.copy()
currentScore = evaluateSolution(currentSolution, rule, prevMonthSchedules, shiftRequests)
bestScore = currentScore
temperature = INITIAL_TEMPERATURE

for iteration in range(MAX_ITERATIONS):
    neighborSolution = generateNeighborSolution(currentSolution)
    neighborScore = evaluateSolution(neighborSolution, rule, prevMonthSchedules, shiftRequests)

    if acceptSolution(currentScore, neighborScore, temperature):
        currentSolution = neighborSolution
        currentScore = neighborScore
        if currentScore < bestScore:
            bestSolution = currentSolution.copy()
            bestScore = currentScore
            noImprovementCount = 0
        else:
            noImprovementCount += 1
    else:
        noImprovementCount += 1

    temperature *= COOLING_RATE

    if noImprovementCount > MAX_NO_IMPROVEMENT:
        temperature = INITIAL_TEMPERATURE  # 재가열
        noImprovementCount = 0

return bestSolution
```

## 8. 제약 조건 상세 명세

> PDF 알고리즘 기반 7가지 규칙 전체 반영

### 8.1 하드 제약 (Hard Constraints) — 위반 시 스케줄 무효

| ID | 규칙 | 설명 | 기본값 | 커스터마이징 |
|----|------|------|--------|------------|
| HC-1 | 근무 유형별 최소 인원 충족 | 매일 각 근무 유형(D/E/N)에 최소 인원이 배정되어야 한다 | 평일: D=3, E=2, N=2 / 주말: D=2, E=2, N=2 | 병동별 설정 가능 |
| HC-2 | 최대 연속 근무일 제한 | 연속 근무일이 상한을 초과할 수 없다 | 5일 | 병동별 설정 가능 |
| HC-3 | 금지 전환 패턴 | 특정 근무 전환 순서를 금지한다 | N→D 금지, N→E 금지, NOD 금지 | 개별 토글 ON/OFF |
| HC-4 | 나이트 근무 보호 | 나이트 단독 근무(1일만 N) 또는 연속 4일 이상 나이트 금지 | 단독 금지, 4일 이상 연속 금지 | 병동별 설정 가능 |

**HC-4 상세:**
- 나이트 단독 근무 금지: 생활 리듬 보호를 위해 N이 1일만 배정되는 것을 방지
- 연속 4일 이상 나이트 금지: 야간 근무 연속 배정을 최대 3일로 제한

### 8.2 소프트 제약 (Soft Constraints) — 위반 시 페널티 부과

| ID | 규칙 | 설명 | 기본값 | 가중치 |
|----|------|------|--------|--------|
| SC-1 | 개인 근무 요청 반영 | 간호사의 특정 날짜 근무/휴무 요청을 반영한다 | 인당 월 최대 2건 | 중간 |
| SC-2 | 균등한 업무 유형 배분 | 간호사 간 D/E/N/O 횟수 편차를 최소화한다 | - | 높음 |
| SC-2a | 주말 근무 횟수 균등 | 주말/공휴일 근무 횟수를 간호사 간 균등 분배 | - | 높음 |
| SC-2b | 숙련도별 조 구성 | 각 근무조에 숙련도가 다른 간호사가 골고루 배치되도록 한다 | - | 중간 |
| SC-2c | 연속 야간 근무 최소화 | 연속 야간 근무 횟수를 가능한 줄인다 (HC-4 이내에서) | - | 중간 |
| SC-3 | 휴무일 연속 배치 보너스 | 휴무일이 연속으로 배치되면 소량의 가산점을 부여한다 | - | 낮음 |

### 8.3 가중치 체계
- 각 제약 조건의 가중치는 **병동별로 독립 설정** 가능
- 하드 제약의 페널티 점수는 소프트 제약 대비 충분히 크게 설정 (예: 10배 이상)
- 가중치 조정 UI를 통해 관리자가 병동 특성에 맞게 튜닝 가능

```
가중치 스케일 예시:
- 하드 제약 위반: 1000점 이상 (사실상 무효화)
- 소프트 제약 (높음): 50~100점
- 소프트 제약 (중간): 20~50점
- 소프트 제약 (낮음): 5~20점
```

## 9. UI/UX 설계 요구사항

### 9.1 전체 화면 구성

| 화면 | 사용자 | 설명 |
|------|--------|------|
| 로그인 | 공통 | 이메일/비밀번호 인증 |
| 대시보드 | 관리자 | 병동 현황, 근무표 상태 요약, 빠른 액션 |
| 병동/간호사 관리 | 관리자 | 간호사 등록, 숙련도 설정, 그룹 관리 |
| 근무 규칙 설정 | 관리자 | 제약 조건 설정, 전환 규칙 토글, 가중치 조정 |
| 근무표 생성 | 관리자 | 기본 설정 입력 → 자동 생성 실행 |
| 근무표 조회/수정 | 관리자 | 테이블 뷰, 셀 수정, 위반 경고, 확정 |
| 통계 대시보드 | 관리자 | 공정성 지표, 근무 분포, 요청 반영률 |
| 내 근무 일정 | 간호사 | 월별/주별 개인 근무표, 개인 일정 병행 |
| 근무 요청 | 간호사 | 휴무/근무 선호 요청 제출 |

### 9.2 근무표 테이블 뷰 (핵심 화면)

**레이아웃:**
- 행(Row): 간호사 이름 (고정 열)
- 열(Column): 날짜 (1일~말일, 요일 표시)
- 셀(Cell): 근무 유형 코드 (D/E/N/O/X) + 색상 코딩

**색상 코딩 (references.md 기반):**
- D (낮번): 파란색 배경
- E (저녁번): 초록색 배경
- N (밤번): 보라색 배경
- O (휴무): 회색 배경
- X (기타): 노란색 배경

**인터랙션:**
- 셀 클릭: 근무 유형 변경 드롭다운 표시
- 변경 시: 실시간 제약 조건 검증 → 위반 시 셀 테두리 빨간색 경고
- 하단 요약: 각 간호사의 D/E/N/O 횟수 집계

### 9.3 근무 규칙 설정 화면 (MakeDuty 참고)

**좌측 패널 — 기본 설정:**
- 스케줄 대상 연월 선택
- 총 간호사 수 표시
- 근무 전환 규칙 토글 (N→D, N→E, ED, NOD 등)

**우측 패널 — 상세 규칙:**
- 최대 연속 근무일 수 (숫자 입력)
- 최대 연속 야간근무 수 (숫자 입력)
- 월간 최소 휴무일 수 (숫자 입력)
- 평일 근무 유형별 인원 (D/E/N 각각)
- 주말 근무 유형별 인원 (D/E/N 각각)

### 9.4 반응형 디자인
- 데스크톱: 2패널 레이아웃, 풀 테이블 뷰
- 태블릿: 단일 패널, 스크롤 테이블
- 모바일: 개인 일정 뷰 중심, 테이블은 가로 스크롤

### 10.1 ERD (주요 엔티티 및 관계)

```
Ward (병동)
├── id (PK)
├── name
├── created_at
└── updated_at

Nurse (간호사)
├── id (PK)
├── ward_id (FK → Ward)
├── name
├── email
├── role (ADMIN | NURSE)
├── skill_level (JUNIOR | SENIOR | CHARGE)
├── password_hash
├── created_at
└── updated_at

Rule (근무 규칙)
├── id (PK)
├── ward_id (FK → Ward, UNIQUE)
├── max_consecutive_work_days (default: 5)
├── max_consecutive_night_shifts (default: 3)
├── min_monthly_off_days
├── weekday_day_count (default: 3)
├── weekday_evening_count (default: 2)
├── weekday_night_count (default: 2)
├── weekend_day_count (default: 2)
├── weekend_evening_count (default: 2)
├── weekend_night_count (default: 2)
├── allow_night_to_day (default: false)
├── allow_night_to_evening (default: false)
├── allow_night_off_day (default: false)
├── constraint_weights (JSON — 제약별 가중치)
├── created_at
└── updated_at

Schedule (근무표)
├── id (PK)
├── ward_id (FK → Ward)
├── year_month (YYYY-MM)
├── status (DRAFT | CONFIRMED)
├── created_at
└── updated_at

ScheduleEntry (근무표 셀)
├── id (PK)
├── schedule_id (FK → Schedule)
├── nurse_id (FK → Nurse)
├── date
├── shift_type (D | E | N | O | X)
└── is_manually_edited (boolean)

ShiftRequest (근무 요청)
├── id (PK)
├── nurse_id (FK → Nurse)
├── schedule_id (FK → Schedule)
├── date
├── requested_shift_type (D | E | N | O | X)
├── status (PENDING | APPROVED | REJECTED)
├── reason (optional)
├── created_at
└── updated_at

Holiday (공휴일/특수일)
├── id (PK)
├── ward_id (FK → Ward)
├── date
├── name
└── type (HOLIDAY | EDUCATION | OTHER)
```

### 10.2 엔티티 관계
- Ward 1:N Nurse (한 병동에 여러 간호사)
- Ward 1:1 Rule (병동당 하나의 규칙 세트)
- Ward 1:N Schedule (병동별 월간 근무표)
- Schedule 1:N ScheduleEntry (근무표는 여러 셀로 구성)
- Nurse 1:N ScheduleEntry (간호사별 여러 날짜 배정)
- Nurse 1:N ShiftRequest (간호사별 여러 요청)
- Schedule 1:N ShiftRequest (근무표별 여러 요청)
